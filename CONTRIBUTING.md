# Contributing

Thank you for choosing to contribute to the project, we really appreciate your time and effort!

## Code of Conduct

This project adheres to the Adobe [Code of Conduct](CODE_OF_CONDUCT.md). By participating, you are expected to uphold
this code. Please report unacceptable behavior to the team.

## Ways to Contribute

There are many forms of contributions. New features, changes to existing features, tests, documentation, bug fixes, or
just good suggestions. For any contribution to be considered, a related [issue](#issue-report-guidelines) is always
required.

The team reviews all issues and contributions submitted by the community. During the review we might require
clarifications from the contributor. If there is no response from the contributor within a reasonable time, the issue
may be closed.

Once a contribution is being reviewed, the team will apply a relevant label to the associated issue.

## Reporting and Fixing

#### Reporting a Bug

1. Visit our [issue tracker](https://github.com/merkle-open/aem-utils/issues).
2. File a `New Issue` as a `Bug Report`.
3. Ensure your issue follows the [issue report guidelines](#issue-report-guidelines).

#### Fixing a Bug

If you have a fix ready for a bug, submit a [pull request](#contributing-features) and reference it in the associated
issue.

## Contributing Features

Creating a pull request involves creating a fork of the project in your personal space, adding your new code in a
branch,
and triggering a pull request.

- Check the
  GitHub [Contributing to a project](https://docs.github.com/en/get-started/exploring-projects-on-github/contributing-to-a-project)
  article on how to fork a repository.
- Check the GitHub [About pull requests](https://help.github.com/articles/using-pull-requests) article on how to perform
  pull requests.

Please be advised that your fork must be based on the `develop` branch and all pull requests must be associated with an
existing issue and target the `develop` branch. Forks based of and pull requests targeting the `master` branch will be
rejected.

## Issue Report Guidelines

A well defined issue report will help in quickly understanding and replicating the problem faced, or the feature
requested.

See our [bug report](.github/ISSUE_TEMPLATE/BUG_TEMPLATE.md)
or [feature request](.github/ISSUE_TEMPLATE/FEATURE_REQUEST_TEMPLATE.md) tempaltes for details on what is expected to be
described. The same information is available when creating a new issue on GitHub.
